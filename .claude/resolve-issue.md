# Issue解決ガイドライン

このファイルは、Claude Issue Auto-resolverから依頼されたIssueを解決するためのガイドラインです。

## 1. Issue種別の判定とラベル付与

まず、Issueに `type:*` ラベルが付与されているか確認してください。

**ラベルが付いていない場合:**
Issueのタイトルと本文を読んで、以下の6種類から適切な種別を判定し、対応するラベルを付与してください：

- `type:bug` - バグ、エラー、不具合の修正
- `type:feature` - 新機能の追加
- `type:investigation` - 調査、分析タスク
- `type:refactoring` - コードのリファクタリング
- `type:documentation` - ドキュメントの更新
- `type:enhancement` - 既存機能の改善

判定が難しい場合は、デフォルトで `type:feature` を付与してください。

## 2. 種別ごとの解決方法

### type:bug - バグ修正

1. バグの内容を確認し、再現手順を理解する
2. 関連するコードを調査して原因を特定する
3. 修正方法を検討し、実装する
4. 修正PRを作成する

### type:feature - 新機能追加

1. 要求される機能の仕様を理解する
2. 実装方針を検討する（必要に応じて複数の選択肢を提示）
3. 既存のコードベースのパターンに従って実装する
4. 実装PRを作成する

### type:investigation - 調査タスク

1. 調査対象を明確化する
2. 関連するコード、ドキュメント、履歴を調査する
3. 調査結果を整理する
4. Issueのコメントに調査結果を簡潔にまとめて投稿する

### type:refactoring - リファクタリング

1. リファクタリング対象のコードを確認する
2. 改善方針を検討する
3. 既存の機能を維持しながらコードを整理する
4. リファクタリングPRを作成する

### type:documentation - ドキュメント更新

1. 更新が必要なドキュメントを確認する
2. 正確な情報を収集する
3. ドキュメントを更新する
4. 読みやすさと正確性を確認する
5. ドキュメント更新PRを作成する

### type:enhancement - 既存機能の改善

1. 改善対象の機能を確認する
2. 改善方針を検討する（必要に応じて複数の選択肢を提示）
3. 既存のコードベースのパターンに従って実装する
4. 改善PRを作成する

## 3. 共通の注意事項

- 実装前に既存のコードパターンを確認し、一貫性を保つこと
- 過度な機能追加や不要なリファクタリングは避けること
- コミットメッセージは明確で簡潔にすること
- PR作成時は適切な説明を含めること
- 実装ワークフロー（implementation-workflow）に従ってブランチ管理を行うこと
